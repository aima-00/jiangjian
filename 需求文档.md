# 多用户聊天系统需求文档

## 项目概述

本文档定义了一个多用户聊天系统的功能需求和非功能需求。该系统基于Linux系统编程，使用C语言实现，支持多用户同时在线聊天。

## 项目背景

随着网络通信技术的发展，即时通讯成为人们日常生活中不可或缺的一部分。然而，基于微信或QQ等大型平台的聊天室数据总是存在泄漏的风险。为了保护用户隐私和数据安全，本项目旨在构建一个自有的聊天系统，让用户能够拥有一个安全可靠的沟通渠道。

## 设计理念

- **模块化设计**：各模块清晰可解耦，便于长期跟进和维护
- **用户体验友好**：对非技术人员体验友好，操作简便，不低于QQ或微信的易用性

## 项目目标

- 实现一个基于TCP协议的聊天服务器
- 支持多个客户端同时连接
- 提供用户认证功能（注册/登录）
- 实现消息实时广播功能
- 确保系统稳定性和安全性

## 初始功能目标

### 4.1 群聊功能
- 实现最多8人的群聊功能
- 支持文字消息发送与接收
- 支持语音消息发送与接收
- 支持图片分享功能
- 支持视频通话功能

### 4.2 用户管理

#### 4.2.1 用户注册
- 系统应允许新用户注册账户
- 用户名必须唯一
- 密码需进行加密存储
- 注册成功后自动登录

#### 4.2.2 用户登录
- 系统应验证用户提供的凭据
- 登录成功后建立聊天会话
- 登录失败应给出明确提示

### 4.3 聊天功能

#### 4.3.1 消息发送
- 用户可以发送文本消息
- 消息应包含发送者标识
- 支持特殊命令（如退出）

#### 4.3.2 消息接收
- 用户能够实时接收其他用户发送的消息
- 消息显示应清晰区分不同用户
- 消息接收不应影响消息发送

### 4.4 系统管理

#### 4.4.1 连接管理
- 服务器应管理多个客户端连接
- 连接断开时应正确清理资源
- 应限制最大连接数

## 非功能需求

### 5.1 性能需求
- 支持至少8个并发连接（群聊人数上限）
- 消息延迟不超过1 second
- 服务器应能持续运行

### 5.2 安全需求
- 用户密码应加密存储
- 防止SQL注入等常见攻击
- 确保数据传输的安全性
- 用户隐私数据得到充分保护

### 5.3 可靠性需求
- 服务器崩溃后应能恢复
- 异常连接应能正确处理
- 数据丢失率应接近零

### 5.4 易用性需求
- 用户界面应简洁直观
- 错误信息应清晰易懂
- 用户应能快速上手使用
- 操作流程应尽量简化

## 系统架构需求

### 6.1 服务端架构
- 使用TCP套接字编程
- 实现多线程处理客户端连接
- 使用互斥锁确保线程安全
- 模块化设计，便于扩展

### 6.2 客户端架构
- 简洁的图形用户界面（使用Qt框架）
- 支持多线程收发消息
- 友好的用户交互体验
- 模块化设计，便于扩展

## 技术需求

### 7.1 开发语言
- **C语言（核心部分）**：用于底层网络通信和系统级操作
- **C++（中间层）**：用于多媒体处理和复杂逻辑
- **Python（脚本工具）**：用于自动化测试和辅助工具
- **JavaScript（前端UI）**：用于实现现代化的图形界面

### 7.2 开发与部署平台
- **开发环境**：Linux操作系统
- **部署环境**：Linux服务器
- **原因**：
  - Linux在处理大量并发连接方面有显著优势
  - Linux提供了高效的I/O多路复用机制（如epoll）
  - Linux的线程和进程管理相对轻量，适合高并发场景
  - Linux的网络栈经过优化，适合网络应用开发
  - Linux的系统调用效率较高，有利于性能优化
  - Linux是开源系统，降低了部署成本
  - Linux提供了更好的资源隔离机制，适合服务端应用

### 7.3 不选择Windows系统的原因
- **API兼容性**：项目深度依赖Linux/Unix特有的系统调用和API（如epoll I/O多路复用、POSIX线程库pthread、特定网络头文件等），无法直接跨平台运行至Windows
- **性能考虑**：Linux在高并发网络应用方面通常表现更优
- **资源效率**：Linux服务器通常比Windows服务器占用更少的系统资源
- **系统调用兼容性**：为确保系统调用的一致性和兼容性，开发环境与生产环境保持一致

### 7.4 客户端技术选型

#### 7.4.1 桌面客户端
- **技术栈**：Qt 框架 (C++)
- **支持平台**：Windows、macOS、Linux
- **原因**：
  - Qt 是成熟稳定的跨平台 GUI 框架
  - 与后端 C++ 代码技术栈保持一致
  - 提供原生性能和丰富的 UI 组件
  - 拥有良好的文档和社区支持

#### 7.4.2 移动端实现策略
- **第一阶段**：暂不实现移动端，专注于桌面端开发
- **第二阶段**：根据项目进展和需求，评估移动端实现方案
  - 原生开发（iOS: SwiftUI/UIKit, Android: Jetpack Compose）
  - 跨平台框架（Flutter/React Native）
  - Qt移动端（备选方案）

### 7.5 实施路线
1. **第一阶段**：优先专注于使用 Qt 完成桌面客户端的开发
2. **第二阶段**：根据项目进展和需求，单独评估并决定移动端的技术栈，以保持模块清晰和维护性，避免过早引入多语言栈的复杂性

### 7.6 Web扩展技术选型
- **后期扩展**：采用 oat++ 框架构建独立的 Web API 层或 WebSocket 网关
- **架构**：与现有基于原始套接字的 TCP 服务器并行运行
- **目的**：实现模块化设计和渐进式扩展

### 7.7 依赖库

#### 7.7.1 核心网络库
- `sys/socket.h` - 套接字编程基础
- `pthread.h` - 多线程处理
- `openssl/evp.h` - 加密哈希处理

#### 7.7.2 多媒体处理库
- **音频处理**：PortAudio 或 PulseAudio 用于音频录制和播放
- **视频处理**：FFmpeg 用于视频编码解码
- **图像处理**：libjpeg, libpng 用于图像格式处理

#### 7.7.3 图形界面库
- **GUI框架**：Qt (C++) 用于构建现代化界面
- **Web技术栈**：Electron 或 NW.js 用于跨平台桌面应用（备选）

#### 7.7.4 安全库
- **OpenSSL**：用于密码哈希和数据传输加密
- **libsodium**：用于端到端加密

#### 7.7.5 通信协议
- **WebSocket**：用于实时双向通信
- **RTP/RTCP**：用于音视频流传输
- **STUN/TURN/ICE**：用于NAT穿透

#### 7.7.6 数据存储
- **SQLite**：轻量级本地数据库
- **Redis**：缓存和会话管理

### 7.8 架构模式
- **微服务架构**：将认证、消息、媒体、存储等功能拆分为独立服务
- **事件驱动**：使用异步事件处理提高响应速度
- **插件化设计**：便于功能扩展

### 7.9 部署技术
- **Docker**：容器化部署，便于管理
- **CI/CD**：自动化测试和部署流水线

## 核心需求总结

本项目旨在实现一个聊天系统，满足以下核心需求：

1. **群聊功能**：支持最多8人的群聊，提供文字、语音、图片和音视频通话的聊天功能
2. **客户端技术栈**：采用Qt作为UI交互的技术栈，确保跨平台兼容性
3. **服务端部署**：部署在Linux系统上，基于Linux服务器开发，充分利用Linux在处理高并发连接方面的优势

### 8.1 设计约束
- 必须使用C语言实现（初期）
- 需要遵循Linux系统编程最佳实践

### 8.2 资源约束
- 内存使用应高效
- CPU占用应保持在合理范围
- 最大支持8个并发用户（初期目标）

### 8.3 时间约束
- 项目开发周期
- 服务器响应时间不超过1秒
